## 2. 프로젝트 구조

```
src/main/java/org/example/
├── handler/
│   └── JsonHandler.java       # JSON 파싱 & 파일 저장 핵심 유틸
├── repository/
│   └── JsonRepository.java    # JsonHandler 기반 CRUD 데이터 계층
├── JsonPocMain.java            # JSON POC 실행 진입점
└── CrudApp.java                # CRUD 콘솔 앱 진입점

data/
├── sample.json                 # POC 실행 시 생성
├── items.json                  # POC 실행 시 생성 (배열)
└── records.json                # CRUD 앱이 사용하는 데이터 파일
```

**흐름:**
```
JSON 문자열 입력
    ↓
JsonHandler.parse()   →  JsonNode (파싱)
    ↓
JsonHandler.save()    →  .json 파일 저장
    ↓
JsonRepository        →  CRUD (배열 파일 관리)
    ↓
CrudApp               →  콘솔 인터페이스
```

## 3. JSON POC 실행 및 검증

**POC 목표:** JSON 문자열을 받아 파싱하고, 파일로 저장하고, 파일에서 다시 읽는 핵심 흐름을 검증합니다.

### 실행 명령어

```powershell
cd C:\Reviewer\poc
.\gradlew.bat runPoc --console=plain
```

### 기대 출력

```
=== Step 1. JSON 문자열 파싱 ===
product : 노트북
price   : 1500000
stock   : 10
tags[0] : 전자제품

=== Step 2. JSON 파일 저장 ===
저장 완료: ...\data\sample.json

=== Step 3. JSON 파일 파싱 ===
파일에서 읽은 product : 노트북
파일에서 읽은 price   : 1500000

=== Step 4. JSON 배열 파싱 ===
  id=1   name=사과     price=1000
  id=2   name=바나나   price=500
  id=3   name=오렌지   price=1500

=== Step 5. JSON 배열 파일 저장 ===
저장 완료: ...\data\items.json

=== Step 6. JSON 유효성 검사 ===
정상 JSON  : true
비정상 JSON: false

POC 완료. data/ 디렉토리에서 결과를 확인하세요.
```

### POC 검증 포인트

| 단계 | 확인 항목 |
|------|-----------|
| Step 1 | 파싱한 필드 값이 원본 JSON과 일치하는지 |
| Step 2 | `data/sample.json` 파일이 생성되었는지 |
| Step 3 | 파일에서 읽은 값이 Step 1의 값과 동일한지 |
| Step 4 | 배열의 각 요소가 순서대로 출력되는지 |
| Step 5 | `data/items.json` 파일이 생성되었는지 |
| Step 6 | 유효한 JSON은 `true`, 잘못된 JSON은 `false`인지 |

## 4. CRUD 애플리케이션 테스트

### 실행 명령어

```powershell
.\gradlew.bat run --console=plain
```

### 메뉴 화면

```
══════════════════════════════════════
     JSON 데이터 관리  (CRUD)
══════════════════════════════════════
 1. 전체 조회
 2. ID 조회
 3. 등록  (JSON 입력)
 4. 수정  (JSON 입력)
 5. 삭제
 0. 종료
══════════════════════════════════════
메뉴 선택 >
```

> 최초 실행 시 `data/records.json`이 자동으로 빈 배열(`[]`)로 생성됩니다.

---

### TC-01 데이터 등록 (Create)

**목적:** JSON 문자열을 입력하면 id가 자동 부여되어 파일에 저장되는지 확인합니다.

#### 테스트 1-1: 첫 번째 데이터 등록

메뉴 `3` 선택 후 아래 JSON 입력:

```
입력 > {"name":"홍길동","score":95,"grade":"A"}
```

**기대 결과:**
```
등록 완료! (부여된 ID: 1)
{
  "id" : 1,
  "name" : "홍길동",
  "score" : 95,
  "grade" : "A"
}
```

#### 테스트 1-2: 두 번째 데이터 등록 (ID 자동 증가 확인)

```
입력 > {"name":"김영희","score":87,"grade":"B"}
```

**기대 결과:** ID가 `2`로 부여됩니다.

#### 테스트 1-3: 필드 구조가 다른 데이터 등록

JSON 구조는 자유롭게 입력 가능합니다.

```
입력 > {"product":"노트북","price":1500000,"brand":"Samsung"}
```

**기대 결과:** ID `3`이 부여되고, 앞의 데이터와 다른 필드 구조로 저장됩니다.

---

### TC-02 전체 조회 (Read - 전체)

**목적:** 저장된 모든 레코드가 출력되는지 확인합니다.

메뉴 `1` 선택

**기대 결과 (TC-01 이후 기준):**
```
[전체 목록 - 3건]
{
  "id" : 1,
  "name" : "홍길동",
  ...
}
{
  "id" : 2,
  ...
}
{
  "id" : 3,
  ...
}
```

#### 데이터가 없을 때 조회

`data/records.json` 내용을 `[]`로 초기화한 뒤 메뉴 `1` 선택:

```
저장된 데이터가 없습니다.
```

---

### TC-03 ID 조회 (Read - ID)

메뉴 `2` 선택

#### 테스트 3-1: 존재하는 ID 조회

```
조회할 ID > 1
```

**기대 결과:** ID 1 레코드만 출력됩니다.

#### 테스트 3-2: 존재하지 않는 ID 조회

```
조회할 ID > 999
```

**기대 결과:**
```
ID 999 에 해당하는 데이터가 없습니다.
```

---

### TC-04 데이터 수정 (Update)

**목적:** 특정 ID의 데이터를 새 JSON으로 교체하되, id는 유지되는지 확인합니다.

메뉴 `4` 선택

```
수정할 ID > 1
```

현재 데이터가 출력된 뒤 새 JSON 입력:

```
입력 > {"name":"홍길동","score":100,"grade":"A+","comment":"만점"}
```

**기대 결과:**
```
수정 완료!
```

**검증:** 메뉴 `2`에서 ID `1` 조회 → `score`가 `100`, `grade`가 `A+`으로 변경되고, `id`는 `1`로 유지되는지 확인합니다.

---

### TC-05 데이터 삭제 (Delete)

메뉴 `5` 선택

#### 테스트 5-1: 정상 삭제

```
삭제할 ID > 2
정말 삭제하시겠습니까? (y/n) > y
```

**기대 결과:**
```
삭제 완료!
```

**검증:** 메뉴 `1`(전체 조회)에서 ID 2가 목록에 없는지 확인합니다.

#### 테스트 5-2: 삭제 취소

```
삭제할 ID > 1
정말 삭제하시겠습니까? (y/n) > n
```

**기대 결과:**
```
삭제가 취소되었습니다.
```

**검증:** ID 1 데이터가 그대로 남아있는지 확인합니다.

## 5. 예외 처리 테스트

| 상황 | 입력 | 기대 결과 |
|------|------|-----------|
| ID 입력란에 문자 입력 | `abc` | `[오류] 숫자를 입력해주세요.` 후 메뉴 복귀 |
| 잘못된 JSON 형식으로 등록 | `{name: 홍길동}` | `[오류] 올바른 JSON 형식이 아닙니다.` 후 메뉴 복귀 |
| 잘못된 JSON 형식으로 수정 | `not json` | `[오류] 올바른 JSON 형식이 아닙니다.` 후 메뉴 복귀 |
| 존재하지 않는 ID 수정 | ID `999` | `ID 999 에 해당하는 데이터가 없습니다.` |
| 존재하지 않는 ID 삭제 | ID `999` | `ID 999 에 해당하는 데이터가 없습니다.` |
| 잘못된 메뉴 번호 | `9` | `[오류] 올바른 번호를 입력하세요.` 후 메뉴 재표시 |

## 6. JSON 파일 직접 확인

### 파일 위치

| 파일 | 생성 시점 | 내용 |
|------|-----------|------|
| `data/sample.json` | POC 실행 시 | 단일 JSON 객체 |
| `data/items.json` | POC 실행 시 | JSON 배열 |
| `data/records.json` | CRUD 앱 최초 실행 시 | CRUD 앱의 데이터 저장소 |

### `data/records.json` 예시 (2건 등록 후)

```json
[ {
  "id" : 1,
  "name" : "홍길동",
  "score" : 95,
  "grade" : "A"
}, {
  "id" : 2,
  "name" : "김영희",
  "score" : 87,
  "grade" : "B"
} ]
```

### 파일로 검증해야 할 항목

| 동작 | 파일에서 확인할 내용 |
|------|---------------------|
| Create | 배열에 새 객체가 추가되고 id가 부여되었는지 |
| Update | 해당 id 객체의 내용이 교체되고 id는 유지되는지 |
| Delete | 해당 id 객체가 배열에서 제거되었는지 |
| 앱 재시작 | 파일 내용이 그대로 유지되는지 |

## 7. 전체 흐름 통합 테스트

```
Step 1.  POC 실행
         .\gradlew.bat runPoc --console=plain
         → data/sample.json, data/items.json 생성 확인

Step 2.  CRUD 앱 실행
         .\gradlew.bat run --console=plain
         → data/records.json 자동 생성 확인

Step 3.  [Create] 메뉴 3
         입력: {"name":"이순신","score":100}
         → "등록 완료! (부여된 ID: 1)" 확인

Step 4.  [Create] 메뉴 3
         입력: {"name":"강감찬","score":75}
         → "등록 완료! (부여된 ID: 2)" 확인

Step 5.  [Read 전체] 메뉴 1
         → 2건 출력 확인

Step 6.  [Read ID] 메뉴 2, ID: 1
         → 이순신 데이터 출력 확인

Step 7.  [Update] 메뉴 4, ID: 1
         입력: {"name":"이순신","score":99,"comment":"수정됨"}
         → "수정 완료!" 확인
         → 메뉴 2로 ID 1 재조회 → score=99, id=1 유지 확인

Step 8.  [Delete] 메뉴 5, ID: 2, y
         → "삭제 완료!" 확인

Step 9.  [Read 전체] 메뉴 1
         → 1건만 출력, ID 2 없는 것 확인

Step 10. 메뉴 0 종료 → 재실행 → 메뉴 1
         → 데이터가 그대로 유지되는지 영속성 확인
```

## 8. 테스트 체크리스트

### JSON POC
- [ ] JSON 문자열 파싱 후 필드 값이 정확한지
- [ ] `data/sample.json` 파일이 생성되는지
- [ ] 저장된 파일을 다시 파싱했을 때 원본과 동일한지
- [ ] JSON 배열 파싱이 정상 동작하는지
- [ ] 유효하지 않은 JSON에 대해 `false`를 반환하는지

### CRUD - Create
- [ ] JSON 입력 시 id가 자동 부여되는지
- [ ] 두 번째 등록 시 id가 1 증가하는지
- [ ] `data/records.json`에 데이터가 추가되는지
- [ ] 필드 구조가 다른 JSON도 저장되는지

### CRUD - Read
- [ ] 전체 조회 시 모든 레코드가 출력되는지
- [ ] 데이터 없을 때 안내 메시지가 출력되는지
- [ ] ID 조회 시 해당 레코드만 출력되는지
- [ ] 존재하지 않는 ID 조회 시 안내 메시지가 출력되는지

### CRUD - Update
- [ ] 수정 후 내용이 새 JSON으로 교체되는지
- [ ] 수정 후 id가 유지되는지
- [ ] `data/records.json`에 변경 내용이 반영되는지

### CRUD - Delete
- [ ] `y` 입력 시 데이터가 삭제되는지
- [ ] `n` 입력 시 데이터가 유지되는지
- [ ] `data/records.json`에서 해당 레코드가 제거되는지

### 예외 처리
- [ ] 잘못된 JSON 형식 입력 시 오류 메시지 후 메뉴 복귀하는지
- [ ] ID 입력란에 문자 입력 시 오류 메시지 후 메뉴 복귀하는지
- [ ] 잘못된 메뉴 번호 입력 시 메뉴가 재표시되는지

### 데이터 영속성
- [ ] 앱 종료 후 재실행해도 데이터가 유지되는지
- [ ] JSON 파일이 사람이 읽기 좋은 형태(들여쓰기 포함)로 저장되는지
